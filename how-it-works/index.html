How It Works

One rule engine. One finding. Three surfaces.

What this is

A demo that reads a queue of simulated prescription refill events and gives every prescription a second look before anyone acts on it. The server evaluates each event against federal dispensing rules; the three surfaces — patient, pharmacist, and prescriber — each render the same finding in the language their reader needs.

The rules

Each rule is one function that takes an event and returns a Finding or nothing.

Architecture

  • src/ingestion.py — the deterministic synthetic roster. Same queue every request, so every count on every screen is reproducible.
  • src/rules/ — the engine. One file per rule; adding a rule means writing one function and registering it.
  • src/transform.py — evaluates each event, attaches the findings and a computed summary. Nothing a screen shows is hardcoded.
  • api/events.py — a Python serverless function on Vercel that serves the evaluated queue as JSON.
  • Static pages — patient, pharmacist, and MD surfaces render the same payload with no framework and no build step.

Data honesty

The interface once carried a green badge reading “SECURE U.S. CLINICAL STREAM.” The application uses synthetic records and makes no security claim, so the badge is gone. A gray line on every page states what the data actually is.

Authoritative sources

  • DailyMed (NIH/NLM) — Structured Product Labeling, including pediatric-use sections.
  • eCFR Title 21 — federal refill and registration requirements for controlled substances.
  • Drugs@FDA — approvals and indications behind the synthetic roster.
